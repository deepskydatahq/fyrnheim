## 1. Source Union -- What Belongs in the Public Library?

### Analysis of Each Source Type

| Source Type | Used By | Pattern | Verdict |
|---|---|---|---|
| `BigQuerySource` | transactions, subscriptions | Generic: "read from a table" | **KEEP** -- rename to `TableSource` |
| `DerivedSource` | person | Generic: "entity from identity resolution" | **KEEP** |
| `DerivedEntitySource` | (placeholder) | Generic: "entity from derived logic" | **KEEP** -- but merge with `DerivedSource` |
| `AggregationSource` | account | Generic: "aggregate one entity into another" | **KEEP** |
| `EventAggregationSource` | (touchpoints) | Generic: "aggregate raw events into entity rows" | **KEEP** |
| `UnionSource` | signals | Generic: "union N sources into common schema" | **KEEP** |
| `AnonSourceConfig` | anon | Specific pattern but reusable: "anonymous identifier tracking" | **STRIP** from core, move to contrib/recipes |
| `SignalSourceConfig` | signals (via UnionSource) | Highly timo-specific: signal_type_column, email_column, field_mappings | **STRIP** from core |
| `ProductSourceConfig` | product | Highly timo-specific: id_prefix, product_type, view_count_column | **STRIP** from core |
| `ProductUnionSource` | product | Wrapper around ProductSourceConfig list | **STRIP** from core |

### Decision: Core Source Union

```python
Source = Union[
    TableSource,          # was BigQuerySource -- reads from project/dataset/table
    DerivedSource,        # entity from identity graph resolution
    AggregationSource,    # aggregate from another entity (group by)
    EventAggregationSource,  # aggregate raw events into entity rows
    UnionSource,          # union N sub-sources into common schema
]
```

**Rationale:**

- These five cover the fundamental data sourcing patterns: direct read, derivation, aggregation, event aggregation, and multi-source union.
- `AnonSourceConfig`, `SignalSourceConfig`, `ProductSourceConfig`, and `ProductUnionSource` are all domain-specific *schemas* for `UnionSource` children. They can live in timo-data-stack as custom source types that extend `BaseSourceConfig`. typedata should provide `BaseSourceConfig` as the extensibility point.
- `DerivedEntitySource` and `DerivedSource` overlap conceptually. Merge them into a single `DerivedSource` with an optional `type` discriminator field for different derivation strategies (identity_graph, custom, etc.).

### Rename: BigQuerySource -> TableSource

`BigQuerySource` is misleading because it already supports DuckDB via `duckdb_path`. The actual pattern is "read a table from project/dataset/table coordinates." Rename to `TableSource` with the same fields. The `BaseSourceConfig` base class stays as `BaseSourceConfig` (or rename to `TableSourceBase`) since it provides `project`, `dataset`, `table`, `duckdb_path`, and `read_table()`.

**Migration path in timo-data-stack:** `BigQuerySource = TableSource` alias for backwards compatibility.

## 2. Entity API: required_fields + source Dual API -- Keep or Simplify?

### Current State

Entity has two ways to define its schema:

1. **`required_fields` + `optional_fields`** -- The "contract" pattern. Entity defines what fields it expects, and a `SourceMapping` bridges the actual data source to those fields. Used by the global `transactions` entity.
2. **`source`** -- The "direct source" pattern. Entity points directly at its data source. Used by `person`, `signals`, `account`, `anon`.

The validator enforces: must have `required_fields` OR `source` (at least one).

### Decision: Keep the Dual API

**Both patterns are genuinely useful and represent different architecture levels:**

- **Contract pattern** (`required_fields`) enables multi-tenant/multi-source entities. You define "a transaction must have transaction_id, customer_email, amount_cents" once, then each tenant maps their Lemonsqueezy/Stripe/Shopify source to that contract via `SourceMapping`. This is a powerful pattern for reusable entity definitions.
- **Direct source pattern** (`source`) is simpler and appropriate when an entity has exactly one source or when the source is tightly coupled (identity graphs, aggregations, unions).

**The validator stays as-is:** `required_fields or source` must be present.

**One refinement:** Make the mutual exclusion explicit in documentation but NOT in validation. An entity *could* have both (contract fields plus a default source), which is useful for development workflows where you want to prototype with a direct source then graduate to the contract pattern.

### API Surface for `all_fields`

The current `all_fields` property has a reasonable fallback chain:
1. If `required_fields` is set: return `required_fields + optional_fields`
2. Elif `source` has `.fields`: return `source.fields`
3. Else: raise

This stays. It gives a consistent way to ask "what fields does this entity have?" regardless of which pattern is used.

## 3. LayersConfig: activity vs activities -- Clean Up

### Current State

```python
class LayersConfig(BaseModel):
    prep: PrepLayer | None = None
    dimension: DimensionLayer | None = None
    activity: ActivityLayer | None = None       # Legacy dbt-based
    activities: ActivityConfig | None = None     # NEW: Ibis-based
    snapshot: SnapshotLayer | None = None
    analytics: AnalyticsLayer | None = None
```

Two fields for the same concept:
- `activity` (singular) -- `ActivityLayer` -- generates dbt SQL with `to_sql()`, CTEs, jinja macros
- `activities` (plural) -- `ActivityConfig` -- the new Ibis-based activity layer with trigger types

### Decision: Drop `activity`, keep `activities`, rename to `activity`

**typedata is an Ibis-native library. dbt SQL generation is out of scope.**

- Remove `activity: ActivityLayer | None = None` entirely
- Rename `activities: ActivityConfig` to `activity: ActivityConfig`
- The `ActivityLayer` class (dbt-based) is NOT extracted into typedata. It stays in timo-data-stack for legacy support.
- The `ActivityConfig` class IS extracted (already in story M001-E001-S003).

**Resulting LayersConfig:**

```python
class LayersConfig(BaseModel):
    prep: PrepLayer | None = None
    dimension: DimensionLayer | None = None
    activity: ActivityConfig | None = None      # Ibis-based activity stream
    snapshot: SnapshotLayer | None = None
    analytics: AnalyticsLayer | None = None

    @model_validator(mode="after")
    def validate_at_least_one_layer(self) -> "LayersConfig":
        if not any([self.prep, self.dimension, self.activity,
                    self.snapshot, self.analytics]):
            raise ValueError("At least one layer must be configured")
        return self
```

**Migration in timo-data-stack:** The `signals` entity already uses `activities=ActivityConfig(...)`. After extraction, change to `activity=ActivityConfig(...)`. No other entity uses the old `activity: ActivityLayer`.

## 4. is_internal Flag -- Keep or Strip?

### Current State

```python
is_internal: bool = False  # Flag to hide entity from UI (internal/prep entities)
```

### Decision: Keep `is_internal`

**This is a universally useful metadata flag, not timo-specific.**

Any project with multiple entities will have "internal" or "intermediate" entities that exist for transformation purposes but should not be exposed to end users, dashboards, or data catalogs. Examples:

- Prep/staging entities that feed into dimension entities
- Identity resolution intermediate entities
- Aggregation source entities

The flag costs nothing (a boolean default False), and removing it would force every consumer to reinvent entity visibility logic.

**Keep as-is.** No changes.

## 5. Exact Entity API Surface for the Public Library

### Entity Class

```python
class Entity(BaseModel):
    """Complete entity definition.

    An entity represents a business object (customer, transaction, product)
    with its schema, source, transformation layers, and quality rules.

    Two patterns for defining entity fields:
    - Contract pattern: Set required_fields (+ optional_fields) and connect
      data sources via SourceMapping
    - Direct source pattern: Set source to point at data directly

    At least one of required_fields or source must be provided.
    """

    model_config = {"arbitrary_types_allowed": True}

    # Identity
    name: str = Field(min_length=1, pattern=r"^[a-z_][a-z0-9_]*$")
    description: str = Field(min_length=1)

    # Transformation layers (at least one required)
    layers: LayersConfig

    # Contract fields (the "what this entity expects" pattern)
    required_fields: list[FieldDef] | None = None
    optional_fields: list[FieldDef] | None = None

    # Direct source (the "where data comes from" pattern)
    source: Source | None = None

    # Core computed columns - always generated for this entity
    core_computed: list["ComputedColumn"] | None = None

    # Core measures - aggregation expressions for this entity
    core_measures: list["Measure"] | None = None

    # Visibility
    is_internal: bool = False

    # Quality checks
    quality: Optional["QualityConfig"] = None

    # --- Validators ---

    @model_validator(mode="after")
    def validate_fields_or_source(self) -> "Entity":
        if self.required_fields is None and self.source is None:
            raise ValueError("Entity must have required_fields or source")
        return self

    # --- Properties ---

    @property
    def all_fields(self) -> list[FieldDef]:
        """All entity fields (required + optional, or from source)."""
        if self.required_fields is not None:
            return self.required_fields + (self.optional_fields or [])
        elif self.source and hasattr(self.source, "fields") and self.source.fields:
            return self.source.fields
        raise ValueError("Entity must have required_fields or source with fields")

    @property
    def all_computed_columns(self) -> list["ComputedColumn"]:
        """All computed columns (core + dimension layer)."""
        columns = list(self.core_computed or [])
        if self.layers.dimension and self.layers.dimension.computed_columns:
            columns.extend(self.layers.dimension.computed_columns)
        return columns

    @property
    def all_measures(self) -> list["Measure"]:
        """All measures defined for this entity."""
        return list(self.core_measures or [])

    def get_layer(self, layer_name: str) -> BaseModel | None:
        """Get layer by name."""
        return getattr(self.layers, layer_name, None)

    def has_layer(self, layer_name: str) -> bool:
        """Check if a layer is configured."""
        return self.get_layer(layer_name) is not None
```

### Changes from timo-data-stack Original

| Aspect | timo-data-stack | typedata | Reason |
|---|---|---|---|
| Source union | 7 types including AnonSourceConfig | 5 generic types | Strip domain-specific sources |
| BigQuerySource name | `BigQuerySource` | `TableSource` | Backend-neutral naming |
| DerivedEntitySource | Separate class | Merged into `DerivedSource` | Reduce redundancy |
| LayersConfig.activity | `ActivityLayer` (dbt) | Removed | No dbt in typedata |
| LayersConfig.activities | `ActivityConfig` (Ibis) | Renamed to `activity` | Clean naming |
| Field class name | `Field` | Consider `FieldDef` | Avoid collision with `pydantic.Field` |
| is_internal | Present | Kept | Universally useful |
| TYPE_CHECKING imports | metadata.components, metadata.quality | typedata.components, typedata.quality | New package paths |

### Field vs FieldDef Naming

The current `Field` class name collides with `pydantic.Field` (aliased as `PydanticField` everywhere). Consider renaming to `FieldDef` or `FieldSchema` to avoid the alias dance. This is a minor ergonomics improvement.

**Recommendation:** Rename to `FieldDef`. The import pattern `from typedata.core import Entity, FieldDef, TableSource` reads cleanly.

## 6. BaseSourceConfig as Extension Point

The key architectural decision for extensibility: `BaseSourceConfig` stays as a public base class that users can subclass for custom source types.

```python
class BaseSourceConfig(BaseModel):
    """Base for table-backed sources. Extend this for custom source types."""
    project: str = Field(min_length=1)
    dataset: str = Field(min_length=1)
    table: str = Field(min_length=1)
    duckdb_path: str | None = None

    def read_table(self, conn, backend: str):
        """Read table from backend (BigQuery, DuckDB, etc.)."""
        ...
```

Users in timo-data-stack can then define:

```python
from typedata.core.source import BaseSourceConfig

class SignalSourceConfig(BaseSourceConfig):
    """timo-specific: signal source with field mappings."""
    name: str
    signal_type_column: str | None
    ...
```

This keeps typedata's Source union clean while allowing arbitrary extensions.

## 7. Forward References Strategy

Entity uses `TYPE_CHECKING` for `ComputedColumn`, `Measure`, and `QualityConfig` to avoid circular imports. This pattern carries over to typedata:

```python
from typing import TYPE_CHECKING

if TYPE_CHECKING:
    from typedata.components import ComputedColumn, Measure
    from typedata.quality import QualityConfig
```

Resolution happens via `Entity.model_rebuild()` in the public API module (story M001-E002-S004).

## Summary of Decisions

1. **Source union:** 5 generic types (TableSource, DerivedSource, AggregationSource, EventAggregationSource, UnionSource). Domain-specific sources stripped; BaseSourceConfig exposed for extension.
2. **Dual API:** Keep `required_fields` + `source`. Both patterns serve real use cases.
3. **activity/activities cleanup:** Drop legacy `ActivityLayer`, rename `activities` to `activity` using `ActivityConfig`.
4. **is_internal:** Keep. Universally useful, zero cost.
5. **Entity API:** Nearly identical to original minus dbt artifacts. Rename `Field` to `FieldDef`, `BigQuerySource` to `TableSource`.

    """Complete entity definition.

    An entity represents a business object (customer, transaction, product)
    with its schema, source, transformation layers, and quality rules.

    Two patterns for defining entity fields:
    - Contract pattern: Set required_fields (+ optional_fields) and connect
      data sources via SourceMapping

    At least one of required_fields or source must be provided.
    """

    # Identity
    name: str = PydanticField(min_length=1, pattern=r"^[a-z_][a-z0-9_]*$")
    description: str = PydanticField(min_length=1)

    # Transformation layers (at least one required)
    layers: LayersConfig

    # Contract fields (the "what this entity expects" pattern)
    required_fields: list[FieldDef] | None = None
    optional_fields: list[FieldDef] | None = None
