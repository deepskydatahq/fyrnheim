# Design: M001-E003-S001 -- Extract base generator class and Ibis code generator structure

**Date:** 2026-02-17
**Story:** M001-E003-S001-generator-base
**Status:** Plan

---

## Context

The `timo-data-stack` IbisCodeGenerator (527 lines) generates Python modules containing
Hamilton-decorated functions that produce Ibis table expressions. Each entity gets a
generated `{entity_name}_transforms.py` module with backend-specific source readers
(DuckDB parquet, BigQuery table) and layer transformation functions (prep, dimension,
snapshot, activity, analytics).

This story extracts the class structure and source generation into typedata. The five
design decisions below shape the entire E003 epic and downstream E004 execution engine.

---

## Decision 1: Drop Hamilton, use plain Ibis functions

**Decision:** Remove the Hamilton dependency. Generated code will be plain Python
functions that accept and return `ibis.Table`.

**Rationale:**

- Hamilton's `@config.when` decorator is the *only* Hamilton feature used in the
  generated code. It serves one purpose: backend switching (DuckDB vs BigQuery).
  That is a runtime concern, not a code-generation concern.
- Hamilton adds a non-trivial dependency (hamilton + its transitive deps) for a
  feature we can replace with 5 lines of Python.
- The execution engine (E004-S002) will select the backend *before* calling
  generated code. The generated functions never need to know which backend they
  are running on -- they receive an `ibis.Table` and return an `ibis.Table`.
- Removing Hamilton makes generated code easier to test: just call the function
  with a mock Ibis table. No DAG driver setup needed.

**What changes:**

| Current (Hamilton)                                | typedata (plain Ibis)                            |
| ------------------------------------------------- | ------------------------------------------------ |
| `@config.when(backend="duckdb")`                  | One `source_{name}()` function, backend injected |
| `from hamilton.function_modifiers import config`   | No Hamilton import                               |
| Hamilton driver resolves function graph at runtime | Execution engine calls functions directly         |

---

## Decision 2: Backend switching via a `backend` parameter on source functions

**Decision:** Use a simple parameter-based approach. Source functions accept
a `backend: str` parameter (or a typed literal/enum) plus the `conn: ibis.BaseBackend`
and branch internally.

However, the *preferred* pattern is to push backend selection even further out:
the source function simply accepts `conn: ibis.BaseBackend` and the source
config tells it what to do. The execution engine is responsible for creating
the right connection.

**Concrete design:**

```python
# Generated source function -- single function, no decorator
def source_transactions(conn: ibis.BaseBackend, backend: str) -> ibis.Table:
    """Read transactions from configured source."""
    if backend == "duckdb":
        parquet_path = os.path.expanduser("~/timo-data/transactions/**/*.parquet")
        return conn.read_parquet(parquet_path)
    elif backend == "bigquery":
        return conn.table("transactions", database=("project", "dataset"))
    else:
        raise ValueError(f"Unsupported backend: {backend}")
```

**Why not a factory/registry pattern:**

- A factory pattern (e.g., `SourceReaderFactory.get("duckdb")`) adds abstraction
  without adding value at this stage. There are exactly two backends. A simple `if`
  in the generated code is transparent, debuggable, and easy to grep.
- If a third backend is needed later, refactoring to a registry is straightforward.
  YAGNI applies.

**Alternative considered -- separate files per backend:**

- Generate `transactions_transforms_duckdb.py` and `transactions_transforms_bigquery.py`.
- Rejected: doubles the generated files, and all layer functions (prep, dimension,
  snapshot) are backend-agnostic anyway. Only source functions differ.

## Decision 3: Generated code as importable modules (not callable objects)

**Decision:** The generator writes `.py` files that can be imported as standard
Python modules. Each module exposes named functions. The modules are *not* wrapped
in classes or callable objects.

**Rationale:**

- The E004-S002 acceptance criteria explicitly say: "execute(entity_name) loads
  generated transform module and runs it." This implies `importlib` loading of a
  module, not instantiating an object.
- E003-S004 acceptance criteria say: "Generated file can be imported as a Python
  module without errors."
- Plain modules are the simplest thing that works. They are testable (`import` +
  call function), inspectable (`ast.parse`), and familiar.

**Module structure for a generated `{entity}_transforms.py`:**

```python
"""
transactions entity transformations.

Auto-generated by typedata. DO NOT EDIT.
"""
import os
import ibis


def source_transactions(conn: ibis.BaseBackend, backend: str) -> ibis.Table:
    ...

def prep_transactions(source_transactions: ibis.Table) -> ibis.Table:
    ...

def dim_transactions(prep_transactions: ibis.Table) -> ibis.Table:
    ...

def snapshot_transactions(dim_transactions: ibis.Table) -> ibis.Table:
    ...
```

The execution engine calls these functions in sequence, threading the output of
each into the next. This is the "explicit pipeline" pattern -- no magic resolution,
just function calls.

## Decision 4: Generator API

**Decision:** The minimal public API is:

```python
class IbisCodeGenerator:
    def __init__(self, entity: Entity):
        ...

    def generate_module(self) -> str:
        """Generate complete Python module source code."""
        ...

    def write_module(self, output_dir: Path) -> Path:
        """Write generated module to disk. Returns path to written file."""
        ...
```

Plus a convenience function:

```python
def generate(entity: Entity, output_dir: Path | str = "generated/") -> Path:
    """Generate Ibis transformation module for an entity."""
    gen = IbisCodeGenerator(entity)
    return gen.write_module(output_dir)
```

**Internal methods (private, underscore-prefixed):**

- `_generate_imports() -> str`
- `_generate_source_functions() -> str`
- `_generate_single_source_functions(source) -> str`
- `_generate_union_source_functions(source: UnionSource) -> str`
- `_generate_prep_function() -> str` (S002)
- `_generate_dimension_function() -> str` (S002)
- `_generate_snapshot_function() -> str` (S003)
- `_generate_activity_function() -> str` (future)
- `_generate_analytics_function() -> str` (future)

**Why not `generate(entity) -> str` as a standalone function:**

- The generator has internal state (entity name used across multiple methods).
  A class cleanly encapsulates this. The current timo-data-stack design already
  uses this pattern and it works well.
- The standalone `generate()` convenience function still exists for the common case.

**No base class:**

- The current `SQLGenerator` ABC in `base.py` was designed for multiple generator
  types (dbt, duckdb, sql, ibis). typedata only has the Ibis generator.
- Adding an abstract base class for a single implementation is premature abstraction.
- If we later need a second generator (e.g., SQL for debugging), we can introduce
  a protocol or ABC at that point.

---

## Decision 5: Generate Ibis Python code (keep current approach, refine it)

**Decision:** Continue generating Python source code that uses Ibis expressions.
Do NOT generate raw SQL strings.

**Rationale:**

- The current timo-data-stack generator already generates Ibis Python code (not SQL).
  The `ibis_code_generator.py` produces `.mutate()`, `.select()`, `.filter()`,
  `.group_by()`, `.aggregate()` calls. This is the right approach.
- Ibis expressions are backend-portable. The same `.mutate(x=t.y.cast("int64"))`
  works on DuckDB and BigQuery without modification.
- Generated Ibis code can be `ast.parse()`-d for validation (acceptance criteria).
- Generated Ibis code is readable and debuggable -- users can open the file and
  understand the transformation logic.
- SQL strings embedded in Python are fragile (quoting, dialect differences, no
  type checking).

**Refinement from current approach:**

The current generator has some patterns that embed SQL-like logic in string
templates (e.g., `_translate_filter_expression` parsing SQL fragments). For
typedata, expressions in entity definitions should be valid Ibis Python
expressions from the start. This eliminates the SQL-to-Ibis translation layer.

Example -- entity definition uses Ibis syntax directly:

```python
# In entity definition
computed_columns=[
    ComputedColumn(name="email_hash", expression="email.lower().hash()"),
]
```

The generator binds this to the table variable:

```python
# Generated code
prep = t.mutate(
    email_hash=t.email.lower().hash(),
)
```

This is what the current generator does via `_bind_expression()` and it should
be preserved as-is.

## Scope for S001 specifically

Per the acceptance criteria, S001 extracts:

1. `IbisCodeGenerator.__init__(entity: Entity)` -- constructor
2. `_generate_imports()` -- import block (updated: no Hamilton import)
3. `_generate_source_functions()` -- dispatches to single/union
4. `_generate_single_source_functions(source)` -- DuckDB + BigQuery in one function
5. `_generate_union_source_functions(source: UnionSource)` -- multi-source union

Layer generation methods (`_generate_prep_function`, etc.) are deferred to S002/S003.
The `generate_module()` method should exist in S001 but only call imports + sources.

## Testing approach

Each acceptance criterion maps to a unit test:

1. **Importable:** `from typedata.generators import IbisCodeGenerator`
2. **Constructor:** `IbisCodeGenerator(sample_entity)` does not raise
3. **Imports generation:** `_generate_imports()` output contains `import ibis`,
   does NOT contain `hamilton`
4. **Source functions:** `_generate_source_functions()` output contains
   `conn.read_parquet` (DuckDB) and `conn.table` (BigQuery)
5. **Union source:** Given entity with UnionSource, generated code has per-source
   functions plus a union function

All tests use `ast.parse()` on the generated output to verify syntactic validity.

### Step 2: Implement `IbisCodeGenerator` class

Port from `timo-data-stack/metadata/generators/ibis_code_generator.py` with three
categories of changes applied during extraction:

#### 2a. Constructor (`__init__`)

Copy as-is. Accepts `entity: Entity`, stores `self.entity` and `self.entity_name`.

```python
from pathlib import Path
from typedata.core import Entity
from typedata.core.source import UnionSource

class IbisCodeGenerator:
    """Generate Ibis Python modules from entity definitions."""

    def __init__(self, entity: Entity):
        self.entity = entity
        self.entity_name = entity.name
```

**Change from source:** Docstring drops "Hamilton/" prefix. Imports change from
`metadata.core` to `typedata.core`.

#### 2b. `_generate_imports()` -- drop Hamilton

The generated module header changes:

| Source line                                         | typedata line               |
| --------------------------------------------------- | --------------------------- |
| `from hamilton.function_modifiers import config`    | (removed entirely)         |
| `Auto-generated from src/entities/{name}.py`        | `Auto-generated by typedata. DO NOT EDIT.` |

The method returns a string containing:
- Module docstring (entity name, auto-generated notice)
- `import os`
- `import ibis`
- No Hamilton import

```python
def _generate_imports(self) -> str:
    return f'''"""
{self.entity_name} entity transformations.

Auto-generated by typedata. DO NOT EDIT.
"""

import os
import ibis
'''
```

#### 2c. `_generate_source_functions()` -- dispatch, unchanged logic

Copy the dispatcher as-is. It checks `isinstance(source, UnionSource)` and delegates
to `_generate_single_source_functions` or `_generate_union_source_functions`.

#### 2d. `_generate_single_source_functions(source)` -- merge two functions into one

This is the core Hamilton-removal transformation. The source currently generates two
separate functions with `@config.when` decorators (`__duckdb` and `__bigquery`
suffixed). Replace with a single function that takes `conn` and `backend` parameters,
using `if/elif` branching.

**Source pattern (Hamilton, two functions):**
```python
@config.when(backend="duckdb")
def source_{name}__duckdb(conn: ibis.BaseBackend) -> ibis.Table:
    ...

@config.when(backend="bigquery")
def source_{name}__bigquery(conn: ibis.BaseBackend) -> ibis.Table:
    ...
```

**Target pattern (plain Python, one function):**
```python
def source_{name}(conn: ibis.BaseBackend, backend: str) -> ibis.Table:
    """Read {name} from configured source."""
    if backend == "duckdb":
        parquet_path = os.path.expanduser("{duckdb_path}")
        return conn.read_parquet(parquet_path)
    elif backend == "bigquery":
        return conn.table("{bq_table}", database=("{bq_project}", "{bq_dataset}"))
    else:
        raise ValueError(f"Unsupported backend: {{backend}}")
```

Implementation reads `source.duckdb_path` (with fallback), `source.project`,
`source.dataset`, `source.table` from the source object -- same attribute access
as the original.

#### 2e. `_generate_union_source_functions(source: UnionSource)` -- merge per-source pairs

Same transformation as 2d, applied to each sub-source in `source.sources`. Each
sub-source currently generates a `__duckdb` and `__bigquery` function pair.
Merge each pair into a single function with `if/elif`.

Then the union aggregator function also merges from two (`__duckdb`, `__bigquery`)
into one that accepts `backend: str` and calls the per-source functions.

**Source pattern (per sub-source, two functions):**
```python
@config.when(backend="duckdb")
def source_{entity}_{src_name}__duckdb(conn) -> ibis.Table: ...
@config.when(backend="bigquery")
def source_{entity}_{src_name}__bigquery(conn) -> ibis.Table: ...
```

**Target pattern (per sub-source, one function):**
```python
def source_{entity}_{src_name}(conn: ibis.BaseBackend, backend: str) -> ibis.Table:
    if backend == "duckdb":
        parquet_path = os.path.expanduser("{duckdb_path}")
        raw = conn.read_parquet(parquet_path){filter_line}
        return raw.select(...)
    elif backend == "bigquery":
        raw = conn.table("{table}", database=("{project}", "{dataset}")){filter_line}
        return raw.select(...)
    else:
        raise ValueError(f"Unsupported backend: {{backend}}")
```

**Union aggregator -- source pattern (two functions):**
```python
@config.when(backend="duckdb")
def source_{entity}__duckdb(source_{entity}_{s1}__duckdb: ibis.Table, ...) -> ibis.Table:
    return ibis.union(...)
@config.when(backend="bigquery")
def source_{entity}__bigquery(...) -> ibis.Table:
    return ibis.union(...)
```

**Union aggregator -- target pattern (one function):**
```python
def source_{entity}(conn: ibis.BaseBackend, backend: str) -> ibis.Table:
    """Union all {entity} sources."""
    parts = [
        source_{entity}_{s1}(conn, backend),
        source_{entity}_{s2}(conn, backend),
        ...
    ]
    return ibis.union(*parts)
```

The key simplification: the union function calls its per-source functions directly
rather than receiving them as Hamilton-injected parameters. This is possible because
we no longer rely on Hamilton's DAG resolution.

Port `_translate_filter_expression` as-is -- it is used by union source generation
and has no Hamilton dependency.

#### 2f. `generate_module()` -- S001 scope

The method exists but only calls `_generate_imports()` + `_generate_source_functions()`.
Layer methods are stubbed or omitted entirely (S002/S003 will add them).

```python
def generate_module(self) -> str:
    """Generate complete Python module source code."""
    parts = [
        self._generate_imports(),
        self._generate_source_functions(),
    ]
    return "\n\n".join(p for p in parts if p)
```

#### 2g. `write_module()` and `generate()` convenience function

Port `write_module` as-is (creates dir, writes `{entity_name}_transforms.py`).

Add the module-level convenience function:

```python
def generate(entity: Entity, output_dir: Path | str = "generated/") -> Path:
    """Generate Ibis transformation module for an entity."""
    gen = IbisCodeGenerator(entity)
    return gen.write_module(Path(output_dir))
```

### Step 3: Port `_bind_expression()` helper

Copy as-is. It is a pure string utility with no external dependencies. Needed by
S002/S003 layer methods, but including it now keeps the class complete and avoids
a partial port. It has zero Hamilton or metadata.core coupling.
