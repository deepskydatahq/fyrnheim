# Design: M001-E004-S005 -- End-to-End Integration Test

**Story:** M001-E004-S005 -- End-to-end integration test: define, generate, execute, verify
**Date:** 2026-02-17
**Status:** ready

## 1. Summary

Build a comprehensive end-to-end test that proves the full fyrnheim pipeline works: define a sample entity with PrepLayer + DimensionLayer + quality checks, generate Ibis transformation code, execute on DuckDB with sample parquet data, and verify the output table has correct schema, row counts, computed column values, and quality results. This is the mission validation story -- if these tests pass, M001 is complete.

---

## 2. Design Decisions

### 2.1 Test structure: single large test or multiple focused tests?

**Decision:** Multiple focused tests within a single test class, sharing a common fixture pipeline.

**Rationale:**

A single monolithic test would be fragile and hard to debug -- when it fails you would not know which stage broke. But fully independent tests would duplicate expensive setup (creating parquet, defining entities, running the pipeline) and lose the important property that we are testing a *connected pipeline*.

The timo-data-stack integration tests (e.g., `TestDimSignalsQueryIntegration`, `TestExportMetricsIntegration`) follow exactly this pattern: a shared fixture creates the database/data, and individual test methods verify different aspects of the result. This is the right model.

**Structure:**

```python
class TestEndToEnd:
    """Full pipeline: define -> generate -> execute -> verify."""

    def test_generate_produces_valid_python(self, e2e_pipeline)
    def test_output_table_has_expected_schema(self, e2e_pipeline)
    def test_output_row_count_matches_input(self, e2e_pipeline)
    def test_computed_columns_have_correct_values(self, e2e_pipeline)
    def test_quality_checks_pass(self, e2e_pipeline)
    def test_quality_check_detects_violations(self, e2e_pipeline_with_bad_data)
```

The `e2e_pipeline` fixture runs the full define-generate-execute pipeline once. Each test method inspects a different aspect of the result. This gives us clear failure messages ("schema wrong" vs. "row count wrong" vs. "quality checks failed") while sharing the expensive pipeline execution.

We also add one negative test (`e2e_pipeline_with_bad_data`) to prove quality checks catch real problems, not just that they pass on clean data.

### 2.2 How to create sample parquet data in fixtures

**Decision:** Use pandas DataFrames written to parquet via `df.to_parquet()` inside pytest fixtures, using `tmp_path`.

**Rationale:**

The timo-data-stack conftest.py establishes this exact pattern -- `sample_orders_data` and `sample_subscriptions_data` are both pandas DataFrames built inline. The story handoff hints explicitly say "Create sample parquet with pandas in test fixture." Pandas is already an implicit dependency through ibis-framework[duckdb], so it adds no new dependency.

**Sample data design:**

The fixture creates a `customers` entity with 10 rows covering known edge cases:

### 2.3 What to verify: schema, row counts, computed columns, quality results

**Decision:** Verify all four categories, each as a separate test method.

| Verification | What exactly | Why |
|---|---|---|
| Schema correctness | Output table column names and types match expected set | Proves the generator and executor produced the right shape |
| Row count | `output.count() == 10` (same as input) | Proves no data loss or duplication in the pipeline |
| Computed column values | Spot-check specific rows for email_hash, email_domain, amount_dollars, is_paying | Proves transformations actually computed correctly, not just that columns exist |
| Quality results | NotNull passes on clean data, Unique passes on email_hash, quality runner returns CheckResult objects with `passed=True` | Proves the quality framework integrates with the execution engine |

**Computed column spot-checks (specific values):**

```python
def test_computed_columns_have_correct_values(self, e2e_pipeline):
    result_df = e2e_pipeline.result_table.to_pandas()

    # Row 0: alice@example.com
    alice = result_df[result_df["id"] == 1].iloc[0]
    assert alice["email_domain"] == "example.com"
    assert alice["amount_dollars"] == 0.0
    assert alice["is_paying"] is False  # free plan, 0 cents

    # Row 1: bob@test.org, pro plan, 2900 cents
    bob = result_df[result_df["id"] == 2].iloc[0]
    assert bob["email_domain"] == "test.org"
    assert bob["amount_dollars"] == 29.0
    assert bob["is_paying"] is True
```

This is more reliable than checking all 10 rows -- it catches real computation bugs while remaining readable and maintainable.

### 2.4 Should tests use tmp_path for isolation or a shared test database?

**Decision:** Use `tmp_path` for full isolation. No shared test database.

**Rationale:**

The timo-data-stack conftest.py provides both patterns: `duckdb_memory_conn` (shared in-memory) and `duckdb_file_conn(tmp_path)` (isolated file). The story handoff explicitly says "Use pytest with tmp_path for isolation." For e2e tests that write generated files and create DuckDB tables, isolation is critical because:

1. Tests must not interfere with each other when run in parallel (`pytest-xdist`)
2. Generated Python files could collide if sharing a directory
3. DuckDB file state persisting between tests would create order-dependent failures
4. `tmp_path` auto-cleans, so no test artifacts accumulate

**Isolation structure per test:**

```
tmp_path/
  data/
    customers.parquet          # input fixture
  generated/
    transform_customers.py     # generated by fyrnheim.generate()
  output.duckdb                # created by fyrnheim.run()
```

Each test gets its own `tmp_path`, so each gets its own parquet, generated code, and DuckDB file. Zero chance of cross-contamination.

### 2.5 Test naming and organization in tests/

**Decision:** Follow the timo-data-stack three-tier directory structure with auto-marking.

```
tests/
  __init__.py
  conftest.py              # shared fixtures + auto-marker logic
  unit/
    __init__.py
    ...                    # unit tests from earlier stories
  integration/
    __init__.py
    ...                    # integration tests from earlier stories
  e2e/
    __init__.py
    test_full_pipeline.py  # <-- this story's test file
```

**Naming conventions:**

- **Directory:** `tests/e2e/` -- matches timo-data-stack, auto-marks with `@pytest.mark.e2e`
- **File:** `test_full_pipeline.py` -- describes what it tests (the full pipeline), not the story ID
- **Class:** `TestEndToEnd` -- groups all pipeline verification tests
- **Methods:** `test_<what_is_verified>` -- e.g., `test_output_table_has_expected_schema`

**conftest.py auto-marking** (carried from timo-data-stack pattern):

```python
def pytest_collection_modifyitems(items):
    """Auto-apply markers based on test location."""
    for item in items:
        test_path = str(item.fspath)
        if "/unit/" in test_path:
            item.add_marker(pytest.mark.unit)
        elif "/integration/" in test_path:
            item.add_marker(pytest.mark.integration)
        elif "/e2e/" in test_path:
            item.add_marker(pytest.mark.e2e)
```

This means we never need to manually decorate e2e tests with `@pytest.mark.e2e` -- placement in the directory is sufficient. Users can run `pytest -m e2e` to run only end-to-end tests or `pytest -m "not e2e"` to skip them for fast iteration.

### 2.6 What is the minimum test coverage that proves the mission is complete?

**Decision:** 6 tests in the e2e file, covering each acceptance criterion, plus full test suite pass.

| Test | Acceptance criterion it proves |
|---|---|
| `test_generate_produces_valid_python` | "Test calls generate() and verifies generated file exists and is valid Python" |
| `test_output_table_has_expected_schema` | "Output DuckDB table has expected columns including computed columns" |
| `test_output_row_count_matches_input` | "Output rows match expected count (no data loss)" |
| `test_computed_columns_have_correct_values` | "Output DuckDB table has expected columns including computed columns" (values, not just names) |
| `test_quality_checks_pass` | "Quality checks pass on the output (NotNull, Unique verified)" |
| `test_quality_check_detects_violations` | Proves quality checks are not vacuously true -- they actually catch bad data |

**Full suite requirement:** The final acceptance criterion is "pytest runs all tests (unit + integration + e2e) and they pass." This means running `pytest tests/ -v` and seeing green across all three tiers. This is not a new test -- it is the meta-requirement that everything still works together.

**What is NOT needed:**

- Performance/timing tests -- not part of M001
- Snapshot tests for generated code -- brittle and better handled by the generator stories
- Multi-entity DAG tests -- single entity is sufficient for M001 validation
- BigQuery backend tests -- M001 only targets DuckDB
- CLI tests -- no CLI exists yet

## 3. Test File Structure

```python
"""End-to-end tests for the full fyrnheim pipeline.

Proves: define entity -> generate Ibis code -> execute on DuckDB -> verify results.
This is the mission validation test for M001.
"""

import ast
from dataclasses import dataclass
from pathlib import Path

import pandas as pd
import pytest


# ---------------------------------------------------------------------------
# Data classes for pipeline results (returned by fixtures)
# ---------------------------------------------------------------------------

@dataclass
class PipelineResult:
    """Holds all artifacts from a pipeline run for test inspection."""
    generated_file: Path
    generated_source: str
    result_df: pd.DataFrame
    quality_results: list  # list of CheckResult
    entity: object  # the Entity definition
    input_row_count: int


# ---------------------------------------------------------------------------
# Fixtures
# ---------------------------------------------------------------------------

@pytest.fixture
def sample_customers_parquet(tmp_path):
    """Create sample customer parquet file with 10 rows of known data."""
    df = pd.DataFrame({
        "id": list(range(1, 11)),
        "email": [
            "alice@example.com", "bob@test.org", "carol@example.com",
            "dave@company.io", "eve@example.com", "frank@test.org",
            "grace@company.io", "henry@example.com", "ivy@test.org",
            "jack@company.io",
        ],
        "name": [
            "Alice Smith", "Bob Jones", "Carol White", "Dave Wilson",
            "Eve Taylor", "Frank Brown", "Grace Lee", "Henry Davis",
            "Ivy Miller", "Jack Moore",
        ],
        "created_at": pd.to_datetime([
            "2025-01-15", "2025-02-20", "2025-03-10", "2025-04-05",
            "2025-05-12", "2025-06-18", "2025-07-22", "2025-08-30",
            "2025-09-14", "2025-10-01",
        ]),
        "plan": [
            "free", "pro", "enterprise", "free", "pro",
            "enterprise", "free", "pro", "enterprise", "free",
        ],
        "amount_cents": [0, 2900, 9900, 0, 2900, 9900, 0, 2900, 9900, 0],
    })
    path = tmp_path / "data" / "customers.parquet"
    path.parent.mkdir(parents=True, exist_ok=True)
    df.to_parquet(path, index=False)
    return path, len(df)


@pytest.fixture
def e2e_pipeline(sample_customers_parquet, tmp_path):
    """Run the full pipeline and return all artifacts for inspection.

    Steps:
    1. Define a customers entity inline (PrepLayer + DimensionLayer + quality)
    2. Call fyrnheim.generate() to produce Ibis transformation code
    3. Call fyrnheim.run() to execute on DuckDB with sample parquet
    4. Collect quality check results
    5. Return PipelineResult with everything tests need
    """
    parquet_path, input_row_count = sample_customers_parquet
    generated_dir = tmp_path / "generated"
    generated_dir.mkdir()

    # --- Step 1: Define entity inline ---
    # (Uses fyrnheim public API -- exact imports depend on E001/E002 implementation)
    from fyrnheim.core import Entity, PrepLayer, DimensionLayer, LayersConfig
    from fyrnheim.quality import QualityConfig, NotNull, Unique

    entity = Entity(
        name="customers",
        description="Sample customers for e2e test",
        layers=LayersConfig(
            prep=PrepLayer(
                model_name="prep_customers",
                # PrepLayer transforms: hash email, cast created_at, divide amount
            ),
            dimension=DimensionLayer(
                model_name="dim_customers",
                # DimensionLayer: email_domain, is_paying flag, signup_month
            ),
        ),
        quality=QualityConfig(
            checks=[
                NotNull("email", "id"),
                Unique("email"),
            ]
        ),
    )

    # --- Step 2: Generate ---
    from fyrnheim import generate
    generated_file = generate(entity, output_dir=generated_dir)

    # --- Step 3: Read generated source ---
    generated_source = generated_file.read_text()

    # --- Step 4: Execute ---
    from fyrnheim import run
    run_result = run(
        entity=entity,
        data_dir=parquet_path.parent,
        output_dir=tmp_path / "output",
        backend="duckdb",
    )

    # --- Step 5: Collect results ---
    return PipelineResult(
        generated_file=generated_file,
        generated_source=generated_source,
        result_df=run_result.to_pandas(),
        quality_results=run_result.quality_results,
        entity=entity,
        input_row_count=input_row_count,
    )


# ---------------------------------------------------------------------------
# Tests
# ---------------------------------------------------------------------------

class TestEndToEnd:
    """Full pipeline: define -> generate -> execute -> verify."""

    def test_generate_produces_valid_python(self, e2e_pipeline):
        """Generated file exists, is .py, and parses as valid Python."""
        result = e2e_pipeline
        assert result.generated_file.exists()
        assert result.generated_file.suffix == ".py"

        # Must parse without SyntaxError
        ast.parse(result.generated_source)

        # Must contain a transform function referencing entity name
        assert "customers" in result.generated_source

    def test_output_table_has_expected_schema(self, e2e_pipeline):
        """Output table contains all expected columns including computed ones."""
        result = e2e_pipeline
        columns = set(result.result_df.columns)

        # Original columns should be present
        for col in ["id", "email", "name", "created_at", "plan", "amount_cents"]:
            assert col in columns, f"Missing original column: {col}"

        # Computed columns from PrepLayer + DimensionLayer
        for col in ["email_hash", "email_domain", "amount_dollars", "is_paying"]:
            assert col in columns, f"Missing computed column: {col}"

    def test_output_row_count_matches_input(self, e2e_pipeline):
        """No rows lost or duplicated during transformation."""
        result = e2e_pipeline
        assert len(result.result_df) == result.input_row_count

    def test_computed_columns_have_correct_values(self, e2e_pipeline):
        """Spot-check computed column values for known input rows."""
        df = e2e_pipeline.result_df

        # Alice: free plan, 0 cents, example.com domain
        alice = df[df["id"] == 1].iloc[0]
        assert alice["email_domain"] == "example.com"
        assert alice["amount_dollars"] == pytest.approx(0.0)
        assert alice["is_paying"] is False

        # Bob: pro plan, 2900 cents, test.org domain
        bob = df[df["id"] == 2].iloc[0]
        assert bob["email_domain"] == "test.org"
        assert bob["amount_dollars"] == pytest.approx(29.0)
        assert bob["is_paying"] is True

        # Carol: enterprise plan, 9900 cents
        carol = df[df["id"] == 3].iloc[0]
        assert carol["amount_dollars"] == pytest.approx(99.0)
        assert carol["is_paying"] is True

    def test_quality_checks_pass(self, e2e_pipeline):
        """Quality checks pass on clean data."""
        result = e2e_pipeline
        assert len(result.quality_results) >= 2  # NotNull + Unique at minimum
        for check_result in result.quality_results:
            assert check_result.passed is True, (
                f"Quality check failed: {check_result.check_name}"
            )

    def test_quality_check_detects_violations(
        self, tmp_path
    ):
        """Quality checks correctly detect bad data (not vacuously true)."""
        # Create data with a NULL email to trigger NotNull failure
        df = pd.DataFrame({
            "id": [1, 2],
            "email": ["valid@example.com", None],  # NULL email
            "name": ["Valid", "Bad"],
            "created_at": pd.to_datetime(["2025-01-01", "2025-01-02"]),
            "plan": ["free", "free"],
            "amount_cents": [0, 0],
        })
        path = tmp_path / "bad_data" / "customers.parquet"
        path.parent.mkdir(parents=True, exist_ok=True)
        df.to_parquet(path, index=False)

        # Define same entity, run pipeline, check that NotNull fails
        from fyrnheim.core import Entity, PrepLayer, DimensionLayer, LayersConfig
        from fyrnheim.quality import QualityConfig, NotNull, Unique
        from fyrnheim import generate, run

        entity = Entity(
            name="customers",
            description="Bad data test",
            layers=LayersConfig(
                prep=PrepLayer(model_name="prep_customers"),
                dimension=DimensionLayer(model_name="dim_customers"),
            ),
            quality=QualityConfig(checks=[NotNull("email")]),
        )

        generated_dir = tmp_path / "bad_generated"
        generated_dir.mkdir()
        generate(entity, output_dir=generated_dir)

        run_result = run(
            entity=entity,
            data_dir=path.parent,
            output_dir=tmp_path / "bad_output",
            backend="duckdb",
        )

        # At least one quality check should fail
        failed = [r for r in run_result.quality_results if not r.passed]
        assert len(failed) >= 1, "NotNull check should fail on NULL email"
```

## 4. Fixture Design Details

### 4.1 The `e2e_pipeline` fixture (session-scoped vs function-scoped)

**Decision:** Use function scope (the default), not session scope.

Even though the pipeline is expensive to run, function scope keeps tests isolated. If we need speed later, we can upgrade to `@pytest.fixture(scope="class")` so the pipeline runs once per `TestEndToEnd` class. But for 6 tests with a 10-row parquet file on in-memory DuckDB, the total runtime should be well under 5 seconds. Premature optimization is not worth the risk of shared mutable state.

### 4.2 Entity definition: inline vs file-based

**Decision:** Define the entity inline in the test fixture, not in `examples/`.

The story handoff says "Create entity inline in test (don't depend on examples/)." This is correct because:
- Tests must be self-contained -- breaking an example file should not break tests
- The test entity can be precisely designed to exercise all verification points
- The `examples/` directory is created by S004, not this story

### 4.3 Pandas as a test-only dependency

Pandas is already pulled in by `ibis-framework[duckdb]` (Ibis uses pandas for `.to_pandas()` conversion). We do not need to add it to `[project.optional-dependencies.dev]` explicitly, but we should document this implicit dependency in the test file docstring to avoid confusion if the Ibis dependency changes.

## 7. Open Questions (resolved)

**Q: Should we test the generated code's content beyond `ast.parse()`?**
A: No. Testing specific generated code content couples the e2e test to the generator's output format, which is the generator story's responsibility (E003-S002). The e2e test cares that the generated code *works* (produces correct output when executed), not what it looks like.

**Q: Should we test with multiple entities to prove DAG resolution?**
A: No. M001 requires proving the pipeline works end-to-end for a single entity. Multi-entity DAG resolution is a future concern. Adding it here would couple the e2e test to the dependency resolver implementation.

**Q: Should the `e2e_pipeline` fixture use `fyrnheim.run()` or lower-level `DuckDBExecutor` directly?**
A: Use `fyrnheim.run()`. The public API is what users will call, and the e2e test should exercise the same code path. If `run()` is broken but `DuckDBExecutor` works, that is a real bug the e2e test should catch.

**Q: Should we use `pytest.approx()` for float comparisons?**
A: Yes. `amount_cents / 100` should produce exact results for these specific test values (0, 29.0, 99.0), but using `pytest.approx()` is defensive against floating-point edge cases and is standard pytest practice.

**Q: What happens when earlier stories (S001-S004) change their APIs?**
A: The e2e test will break, which is exactly what we want. It serves as a contract test for the public API. The exact imports and function signatures in section 3 are illustrative -- they will be updated during implementation to match the actual APIs from S001-S004.

@pytest.fixture
def e2e_pipeline(sample_customers_parquet, tmp_path):
    """Run the full pipeline and return all artifacts for inspection.

    Steps:
    1. Define a customers entity inline (PrepLayer + DimensionLayer + quality)
    2. Call fyrnheim.generate() to produce Ibis transformation code
    3. Call fyrnheim.run() to execute on DuckDB with sample parquet
    4. Collect quality check results
    5. Return PipelineResult with everything tests need
    """
    parquet_path, input_row_count = sample_customers_parquet
    generated_dir = tmp_path / "generated"
    generated_dir.mkdir()

    # --- Step 1: Define entity inline ---
    # Uses fyrnheim public API. Exact imports will be adjusted during
    # implementation to match the actual APIs from upstream stories.
    from fyrnheim.components import ComputedColumn
    from fyrnheim.core import DimensionLayer, Entity, LayersConfig, PrepLayer
    from fyrnheim.primitives import date_trunc_month, hash_email
    from fyrnheim.quality import NotNull, QualityConfig, Unique

    entity = Entity(
        name="customers",
        description="Sample customers for e2e test",
        layers=LayersConfig(
            prep=PrepLayer(
                model_name="prep_customers",
                computed_columns=[
                    ComputedColumn(
                        name="email_hash",
                        expression=hash_email("email"),
                        description="SHA256 hash of lowercase trimmed email",
                    ),
                    ComputedColumn(
                        name="amount_dollars",
                        expression="t.amount_cents / 100.0",
                        description="Monthly payment in dollars",
                    ),
                ],
            ),
            dimension=DimensionLayer(
                model_name="dim_customers",
                computed_columns=[
                    ComputedColumn(
                        name="email_domain",
                        expression="t.email.split('@')[1]",
                        description="Email domain extracted from address",
                    ),
                    ComputedColumn(
                        name="is_paying",
                        expression="t.plan != 'free'",
                        description="True if customer is on a paid plan",
                    ),
                    ComputedColumn(
                        name="signup_month",
                        expression=date_trunc_month("created_at"),
                        description="Signup month for cohort analysis",
                    ),
                ],
            ),
        ),
        quality=QualityConfig(
            checks=[
                NotNull("email", "id"),
                Unique("email"),
            ]
        ),
    )

    # --- Step 2: Generate ---
    from fyrnheim import generate

    gen_result = generate(entity, output_dir=generated_dir)

    # generate() returns GenerateResult with .output_path and .code
    generated_file = gen_result.output_path
    generated_source = gen_result.code

    # --- Step 3: Execute ---
    from fyrnheim import run

    run_result = run(
        entity=entity,
        data_dir=parquet_path.parent,
        output_dir=tmp_path / "output",
        backend="duckdb",
    )

    # --- Step 4: Collect results ---
    return PipelineResult(
        generated_file=generated_file,
        generated_source=generated_source,
        result_df=run_result.to_pandas(),
        quality_results=run_result.quality_results,
        entity=entity,
        input_row_count=input_row_count,
    )

class TestEndToEnd:
    """Full pipeline: define -> generate -> execute -> verify."""

    def test_generate_produces_valid_python(self, e2e_pipeline):
        """Generated file exists, is .py, and parses as valid Python.

        AC: Test calls generate() and verifies generated file exists and is valid Python.
        """
        result = e2e_pipeline
        assert result.generated_file.exists()
        assert result.generated_file.suffix == ".py"

        # Must parse without SyntaxError
        ast.parse(result.generated_source)

        # Must contain a transform function referencing entity name
        assert "customers" in result.generated_source

    def test_output_table_has_expected_schema(self, e2e_pipeline):
        """Output table contains all expected columns including computed ones.

        AC: Output DuckDB table has expected columns including computed columns.
        """
        result = e2e_pipeline
        columns = set(result.result_df.columns)

        # Original columns should be present
        for col in ["id", "email", "name", "created_at", "plan", "amount_cents"]:
            assert col in columns, f"Missing original column: {col}"

        # Computed columns from PrepLayer + DimensionLayer
        for col in ["email_hash", "email_domain", "amount_dollars", "is_paying"]:
            assert col in columns, f"Missing computed column: {col}"

    def test_output_row_count_matches_input(self, e2e_pipeline):
        """No rows lost or duplicated during transformation.

        AC: Output rows match expected count (no data loss).
        """
        result = e2e_pipeline
        assert len(result.result_df) == result.input_row_count

    def test_computed_columns_have_correct_values(self, e2e_pipeline):
        """Spot-check computed column values for known input rows.

        AC: Output DuckDB table has expected columns including computed columns
        (verifies values, not just column presence).
        """
        df = e2e_pipeline.result_df

        # Alice: free plan, 0 cents, example.com domain
        alice = df[df["id"] == 1].iloc[0]
        assert alice["email_domain"] == "example.com"
        assert alice["amount_dollars"] == pytest.approx(0.0)
        assert alice["is_paying"] is False

        # Bob: pro plan, 2900 cents, test.org domain
        bob = df[df["id"] == 2].iloc[0]
        assert bob["email_domain"] == "test.org"
        assert bob["amount_dollars"] == pytest.approx(29.0)
        assert bob["is_paying"] is True

        # Carol: enterprise plan, 9900 cents
        carol = df[df["id"] == 3].iloc[0]
        assert carol["amount_dollars"] == pytest.approx(99.0)
        assert carol["is_paying"] is True

    def test_quality_checks_pass(self, e2e_pipeline):
        """Quality checks pass on clean data.

        AC: Quality checks pass on the output (NotNull, Unique verified).
        """
        result = e2e_pipeline
        assert len(result.quality_results) >= 2  # NotNull + Unique at minimum
        for check_result in result.quality_results:
            assert check_result.passed is True, (
                f"Quality check failed: {check_result.check_name}"
            )

    def test_quality_check_detects_violations(self, tmp_path):
        """Quality checks correctly detect bad data (not vacuously true).

        This test uses its own data/entity inline -- it does NOT share the
        e2e_pipeline fixture, because it needs deliberately bad data.
        """
        # Create data with a NULL email to trigger NotNull failure
        df = pd.DataFrame(
            {
                "id": [1, 2],
                "email": ["valid@example.com", None],  # NULL email
                "name": ["Valid", "Bad"],
                "created_at": pd.to_datetime(["2025-01-01", "2025-01-02"]),
                "plan": ["free", "free"],
                "amount_cents": [0, 0],
            }
        )
        path = tmp_path / "bad_data" / "customers.parquet"
        path.parent.mkdir(parents=True, exist_ok=True)
        df.to_parquet(path, index=False)

        # Define same entity structure, run pipeline, check that NotNull fails
        from fyrnheim.core import DimensionLayer, Entity, LayersConfig, PrepLayer
        from fyrnheim.quality import NotNull, QualityConfig

        entity = Entity(
            name="customers",
            description="Bad data test",
            layers=LayersConfig(
                prep=PrepLayer(model_name="prep_customers"),
                dimension=DimensionLayer(model_name="dim_customers"),
            ),
            quality=QualityConfig(checks=[NotNull("email")]),
        )

        from fyrnheim import generate, run

        generated_dir = tmp_path / "bad_generated"
        generated_dir.mkdir()
        generate(entity, output_dir=generated_dir)

        run_result = run(
            entity=entity,
            data_dir=path.parent,
            output_dir=tmp_path / "bad_output",
            backend="duckdb",
        )

        # At least one quality check should fail
        failed = [r for r in run_result.quality_results if not r.passed]
        assert len(failed) >= 1, "NotNull check should fail on NULL email"

### Risk Mitigation

1. **API drift from design docs:** The test code uses illustrative imports based on
   upstream design docs. The implementing agent MUST inspect the actual implemented
   source code and adjust imports, constructor args, and result access patterns.
   This is called out explicitly in Task 4's implementation notes.

2. **Missing upstream stories:** If any upstream story is not yet implemented, the
   e2e test cannot pass. The agent should verify all prerequisites in the table above
   before starting. If a prerequisite is missing, stop and report which stories need
   to be completed first.

3. **`is_paying` boolean type:** DuckDB/pandas may represent booleans as `numpy.bool_`
   rather than Python `bool`. The assertion `alice["is_paying"] is False` may need to
   be `assert alice["is_paying"] == False` or `assert not alice["is_paying"]` depending
   on the actual type. Use `assert not alice["is_paying"]` and
   `assert bob["is_paying"]` (truthy/falsy) as the safer pattern if `is` comparison
   fails.

4. **Fixture sharing vs isolation:** The `e2e_pipeline` fixture is function-scoped
   (the default). This means it runs once per test method that uses it. For 5 tests
   sharing it, the pipeline executes 5 times. If this is too slow, upgrade to
   `@pytest.fixture(scope="class")` on `e2e_pipeline`. But with 10 rows on in-memory
   DuckDB, each run should complete in well under 1 second.
